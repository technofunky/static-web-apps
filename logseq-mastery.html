The Mental Model

Logseq isn't a note-taking app — it's an outliner where everything is a block and every block is addressable. Understanding this changes how you use it.

```
Graph
Your entire knowledge base. All pages and their connections. Stored as plain Markdown files on your disk — you always own your data.
Page
A file. It has a title (the filename). Pages are created automatically when you [[link to them]] — you never need to "create" one first.
Block
Every bullet point. The atomic unit of Logseq. Blocks can be nested, referenced, embedded, queried, and tagged independently. Each block has a unique ID.
Journal
Daily pages auto-created by date. The recommended capture inbox — write everything here first, then link it. Don't organize upfront; let queries surface it.
Graph View
Visual map of connections. Useful for exploration, but not the primary interface. The real power is in linked references and queries, not the graph.
Core philosophy Write on today's journal page. Link liberally to pages (concepts, people, projects). Let the database emerge. Query it later. Never reorganize — just re-link.
Common mistake Treating Logseq like a folder-based note app. Resist creating elaborate folder hierarchies. Pages and links replace folders. Queries replace search.
```

Linking & Tagging Strategy

The difference between a useless graph and a powerful one is intentional linking. Here's the expert system.

```

When to Use Each Link Type

[[Page Link]]
Use for concepts, people, projects, places, books, companies — anything that deserves its own page and will accumulate references over time. The backbone of your graph.
#tag
Use for categories and types you want to filter by. Best for things like #idea #question #resource #insight. These are really just page links with different syntax.
((block ref))
Use when you want to reuse specific content without copy-pasting. The reference stays live — edit the source and all references update. Great for recurring definitions or key quotes.
property::
Use for structured metadata you'll query. If you find yourself writing "Status: In Progress" in prose, make it a property instead. Queryable, consistent, powerful.

Namespaces — The Expert Move

What they are Use / in page names to create hierarchy: [[Books/Deep Work]], [[Projects/Q1 Launch]], [[People/Alice Smith]]. This creates a parent page automatically and lets you navigate hierarchically.
``` # Good namespace patterns [[Books/Atomic Habits]] [[People/John Doe]] [[Projects/Website Redesign]] [[Areas/Health]] [[Resources/Productivity]] # Then query all books: {{query (namespace Books)}}
```

Linked References — The Hidden Feature

At the bottom of every page is a "Linked References" section showing every block that links to this page. This is automatic, zero-effort indexing. It means you never need to manually maintain a list — just write and link.

Power move Create a page called [[Alice Smith]] and never write on it. Just mention [[Alice Smith]] in your journal and meeting notes. Her page becomes an automatic log of every interaction. Same for any project or concept.
```

Queries — Logseq's Superpower

Queries let you surface blocks dynamically based on tags, properties, and content. They turn your graph into a database.

```

Simple Queries

Use {{query ...}} in any block. Simple queries use a basic syntax:

All TODO tasks
{{query (todo todo)}}
All pages with a specific tag
{{query [[project]]}}
Tasks due this week
{{query (and (todo todo doing) (between -7d +7d))}}
All blocks tagged #idea containing "AI"
{{query (and [[idea]] (full-text-search "AI"))}}

Advanced Queries (Datalog)

For true power, use advanced queries in a query code block. These use Datalog syntax:

``` # All books with status = “reading” #+BEGIN_QUERY {:title “Currently Reading” :query [:find (pull ?b [*]) :where [?b :block/properties ?props] [(get ?props :type) ?type] [(= ?type “book”)] [(get ?props :status) ?status] [(= ?status “reading”)]]} #+END_QUERY
```
``` # Tasks scheduled for today #+BEGIN_QUERY {:title “Due Today” :query [:find (pull ?b [*]) :in $ ?start ?end :where [?b :block/scheduled ?d] [(>= ?d ?start)] [(<= ?d ?end)]] :inputs [:today :today] :result-transform (fn [result] (sort-by (fn [r] (get r :block/scheduled)) result))} #+END_QUERY
```
Strategy Don't memorize Datalog. Build a "Query Library" page and save your best queries there. Use block references to reuse them. The key insight: properties you define = columns in your database.

Query Operators Reference

(and ...)All conditions must match
(or ...)Any condition must match
(not ...)Exclude matching blocks
(todo todo doing)Blocks with task state
(between -7d now)Date range filter
(property status "done")Match a property value
(page-property type "book")Match page-level property
```

Expert Workflows

Concrete systems used by power users. Pick one, run it for a month, then add another.

```

Daily Journaling System Beginner

1

Morning capture

Open journal (Alt+J). Log your top 3 priorities as TODOs. Add [[Projects/X]] links to each.

2

Capture throughout the day

Every note, idea, or meeting goes in today's journal. Tag liberally: #idea #question [[Person Name]].

3

Evening review

Mark tasks done. Add a quick reflect:: property with the day's key lesson. Nothing else required.

Book Notes System Intermediate

``` # Template: Books/[Title] type:: book author:: [[Author Name]] status:: reading # → reading → done rating:: - finished:: - - ## Summary - - ## Key Ideas - - ## Highlights - - ## Actions - TODO

Then query all books with a rating: {{query (and (page-property type "book") (page-property rating))}}

Meeting Notes System Intermediate

# On today’s journal page: - Meeting: Q1 Planning attendees:: [[Alice]] [[Bob]] project:: [[Projects/Website]] date:: [[2024-01-15]] - ## Decisions - Launching March 1 - ## Action Items - TODO [[Alice]] to send mockups by Friday - TODO [[Bob]] to review budget
Result Alice's page now shows this meeting in linked references. The project page shows it too. Query {{query (and (todo todo) (property project [[Projects/Website]]))}} to see all open tasks for the project.

GTD Task System Advanced

Create a "Dashboard" page with embedded queries:

NOW / DOING
{{query (todo now doing)}}
Scheduled Today
{{query (and (todo todo) (scheduled :today))}}
Overdue
{{query (and (todo todo) (deadline :before :today))}}
Waiting On
{{query (and (todo todo) [[waiting]])}}

Essential Plugins

Install from Settings → Plugins. These are the ones experts actually use.

🗓

Logseq Calendar

Monthly calendar view with dots on days with journal entries. Instant visual navigation to any past day. Install this first.

📋

Logseq Kanban

Turns any tagged list into a Kanban board. Create boards from type:: kanban pages. Perfect for project management.

🎨

Logseq Diagrams

Draw.io integration built in. Use /draw to create flowcharts, diagrams, and visual maps stored in your graph.

🔗

Random Note

Opens a random page from your graph. Incredible for rediscovering forgotten notes and making unexpected connections. Use daily.

📊

Logseq Charts

Render charts from your data. Define chart data in a code block with type echarts. Good for tracking habits and metrics.

⚡

Quickly Capture

Global hotkey to capture a note to today's journal from anywhere on your computer, without switching to Logseq. Essential for quick capture.

🔎

Smart Blocks

Dynamic templates with variables, conditional blocks, and auto-fill. The most powerful templating system for Logseq power users.

🌙

Logseq Bujo

Bullet Journal inspired views. Adds weekly/monthly log views and future log — great if you like BuJo-style planning.

Plugin philosophy Install one plugin at a time and actually use it for 2 weeks before adding another. Plugin overload kills productivity. The graph and queries are the real power — plugins are multipliers, not foundations.

Your Mastery Path

Work through this checklist. Click items to mark them done. These are ordered by impact — do them in sequence.

Week 1 — Foundations

  • Write in journal every day for 7 days straight
  • Learn Ctrl+K (search) — use it instead of clicking pages
  • Create 10 page links [[like this]] naturally in your writing
  • Add a property (like status::) to 5 blocks
  • Explore Linked References on a page you've mentioned often
  • Create your first template with /template

Week 2-3 — Structure

  • Set up a consistent namespace system (Books/, Projects/, People/)
  • Create a Dashboard page with 3 live queries
  • Write your first advanced query (#+BEGIN_QUERY)
  • Use block references ((uuid)) to link two related blocks
  • Create your book notes template and log 3 books
  • Install and use the Calendar plugin for 1 week

Month 2 — Systems

  • Have 100+ pages in your graph — you've earned the power
  • Build a project management system with queries
  • Set up Smart Blocks for your most-used templates
  • Use Random Note plugin weekly for spaced repetition
  • Write a "meta-note" about how you use Logseq — query it later
  • Configure a syncing solution (Git, iCloud, or Logseq Sync)

Expert Level — Mastery Signs

  • You write first, organize never — queries surface everything
  • Your graph has unexpected connections you didn't plan
  • You can write a Datalog query from scratch in under 5 minutes
  • You've found something in your notes that changed a decision
